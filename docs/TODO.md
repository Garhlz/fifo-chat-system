### 涉及文件

```text

```

### 实现要点

实现 `daemonize()`：

1. 第一次 `fork()`，父进程退出；
2. 子进程调用 `setsid()`；
3. 忽略 `SIGHUP`；
4. 可选第二次 `fork()`；
5. 设置 `umask`；
6. 修改工作目录；
7. 关闭标准输入、标准输出、标准错误；
8. 重定向到 `/dev/null`。

### 验证方式

启动服务器后执行：

```bash
ps -ef | grep chatserver
ps -o pid,ppid,stat,tty,cmd -C chatserver_ynp_1.0
```

确认：

1. 服务器仍在后台运行；
2. `TTY` 为 `?`；
3. 父进程已退出；
4. 日志中记录服务器启动。

### 注意事项

* 调试阶段可以先提供 `--foreground` 参数，允许服务器不 daemonize，方便看输出；
* 最终报告截图时必须展示守护进程状态；
* daemonize 后不能依赖 `printf()` 调试，应使用日志。

## 阶段 8：实现线程池分派和回收

### 目标

实现请求到达后分派 worker，处理完成后回收 worker，并满足“刚回收线程倾向于下一次被分派”。

### 涉及文件

```text
src/thread_pool.c
include/thread_pool.h
src/handlers.c
```

### 实现要点

1. 使用 LIFO 空闲线程栈；
2. 初始化时将所有 worker id 压入空闲栈；
3. 分派时从栈顶弹出 worker id；
4. 将 worker 状态改为 `BUSY`；
5. 将请求复制给 worker；
6. 唤醒 worker；
7. worker 处理完成后调用回收函数；
8. 回收时将 worker 状态改为 `IDLE`；
9. 将 worker id 压回空闲栈；
10. 记录分派和回收日志。

### 验证方式

临时模拟多个请求，让服务器把请求交给线程池。查看：

```bash
cat ~/log/chat-logs/server/threads.log
```

确认：

1. 有 worker 被分派；
2. 有 worker 被回收；
3. 同一个刚回收 worker 在后续请求中优先出现。

### 注意事项

* 操作 `idle_stack` 必须加线程池 mutex；
* worker 的任务字段必须在线程安全条件下赋值；
* 条件变量等待必须放在 while 循环中；
* 日志中要体现 worker id 和请求类型。

## 阶段 9：实现 poll 多路复用监听

### 目标

服务器主线程使用 `poll()` 同时监听注册、登录、发消息、退出 4 个公共 FIFO。

### 涉及文件

```text
src/server.c
src/fifo.c
include/chat.h
include/thread_pool.h
```

### 实现要点

1. 打开 4 个公共 FIFO；
2. 建立 `pollfd fds[4]`；
3. 每个 fd 的 `events` 设置为 `POLLIN`；
4. 调用 `poll(fds, 4, -1)` 阻塞等待请求；
5. 检查 `revents`；
6. 如果某个 FIFO 可读，则读取 `chat_request_t`；
7. 读到完整请求后交给线程池；
8. 记录请求到达日志；
9. 处理 `EINTR`；
10. 处理非阻塞读可能出现的 `EAGAIN`。

### 验证方式

写一个临时客户端或使用简单测试程序向 4 个 FIFO 写入请求。

查看：

```bash
cat ~/log/chat-logs/server/server.log
cat ~/log/chat-logs/server/threads.log
```

确认服务器能监听到不同 FIFO 的请求，并分派给 worker。

### 注意事项

* FIFO 建议用 `O_RDWR | O_NONBLOCK` 打开；
* 每次读取必须判断是否读满一个结构体；
* 不完整请求应记录错误；
* `poll()` 的 fd 集合在本实验中基本固定，变化主要体现在 `revents`。

* 推送离线消息时，注意不要长时间持有 user_store 锁进行阻塞写 FIFO；
* 可以先复制待推送消息，再释放锁写 FIFO。

## 阶段 13：实现客户端基础框架

### 目标

实现普通客户端 `client`，支持注册、登录、发送消息、退出和接收消息。

### 涉及文件

src/client.c
include/chat.h

客户端需要支持命令行或交互式输入。

send 目标用户名 消息内容

* 客户端退出时要清理自己的 FIFO；

* 客户端和服务器使用同一份 `chat.h`。

## 阶段 14：实现在线消息发送

### 目标

实现用户之间的即时消息转发。

### 涉及文件

```text
src/handlers.c
src/user_store.c
src/client.c
src/log.c
```

### 实现要点

消息处理流程：

1. worker 收到 `REQ_MSG`；
2. 检查发送者是否在线；
3. 检查接收者是否存在；
4. 如果接收者在线：

   * 获取接收者专用 FIFO；
   * 向该 FIFO 写入消息；
   * 记录 `sent` 日志；
   * 向发送者返回发送成功；
5. 如果接收者离线：

   * 进入离线消息逻辑；
6. 如果接收者不存在：

   * 返回失败信息；
   * 记录错误日志。

### 验证方式

`学生姓名拼音_1` 给 `学生姓名拼音_2` 连续发送 5 次：

```text
How are you，学生姓名拼音_2
```

预期：

1. `学生姓名拼音_2` 终端显示 5 条消息；
2. `server.log` 中有 5 条 `sent` 日志；
3. 如果实现重要朋友标记，第 5 次后可在目标用户名后显示 `*`。

### 注意事项

* 写用户 FIFO 时可能阻塞，尽量不要持有 user_store 锁；
* 消息中需要带发送时间；
* 日志要记录发送者、接收者和状态。

## 阶段 16：实现离线消息保存和登录后推送

### 目标

实现用户离线时消息保存，重新登录后收到消息和原始发送时间。

### 涉及文件

```text
src/handlers.c
src/user_store.c
src/client.c
src/log.c
```

### 实现要点

当接收者离线时：

1. 判断接收者是否存在；
2. 如果是普通用户：

   * 保存离线消息；
   * 状态为 `pending`；
   * 记录 pending 日志；
   * 告诉发送者消息已保存；
3. 如果是机器人用户：

   * 不长期保存；
   * 丢弃消息；
   * 记录 `discarded_bot_offline` 日志。

用户登录成功后：

1. 查找该用户 pending 消息；
2. 推送到用户专用 FIFO；
3. 消息中显示原始发送时间；
4. 推送成功后标记为 `sent`；
5. 写入 sent 日志。

### 验证方式

1. `学生姓名拼音_2` 退出；
2. `学生姓名拼音_1` 发送：

```text
Hi，let’s play badminton?
```

3. `server.log` 出现 pending；
4. `学生姓名拼音_2` 重新登录；
5. `学生姓名拼音_2` 收到离线消息和发送时间；
6. `server.log` 出现 sent。

### 注意事项

* 离线消息结构中必须保存发送时间；
* 不要在推送离线消息时重复推送已经 sent 的消息；
* 可以使用数组保存离线消息，实验规模足够；
* 需要考虑离线消息数组满的情况。

## 建议实现顺序总结

推荐按以下顺序实际开发：

```text
1. 项目骨架和 Makefile
2. 公共协议 chat.h
3. 配置读取
4. 日志模块
5. FIFO 创建
6. daemonize
7. 服务器初始化
8. 线程池创建
9. 线程池分派和回收
10. poll 主循环
11. user_store
12. 注册
13. 登录
14. client 基础框架
15. 在线消息
16. 退出广播
17. 离线消息
18. server.log 完善
19. threads.log 完善
20. bot_manager 命令框架
21. 增加机器人
22. 机器人自动回复
23. 减少机器人
24. 完整集成测试
25. 整理截图
26. 整理 prompts
```

按照这个顺序实现，可以先保证服务器底层机制可运行，再逐步补齐业务功能，最后实现机器人管理器。这样比较容易定位问题，也更方便后续撰写实验报告。
