# 系统编程课程实验需求文档：基于 FIFO 的即时聊天系统

## 1. 实验目标

本实验要求使用 C 语言、POSIX 系统调用、标准 C 库函数和 POSIX pthread 库，实现一个基于命名管道 FIFO 的即时聊天系统。

系统需要包含三个主要程序：

1. 服务器程序 `chatserver`
2. 普通客户端程序 `client`
3. 聊天机器人管理器 `bot_manager`

服务器需要作为多线程守护进程运行，主线程通过多路复用技术监听多个公共 FIFO，请求到达后分派给线程池中的工作线程处理。系统需要支持普通用户注册、登录、发送消息、退出、离线消息保存与推送，同时支持动态增加和减少聊天机器人。

---

## 2. 运行环境要求

实验运行环境为：

* openEuler 系统；
* C 语言；
* POSIX 系统调用；
* 标准 C 库；
* POSIX pthread 库；
* FIFO 命名管道；
* `select()` / `poll()` / `epoll()` 三选一进行多路复用，优先采用 `poll()`；
* 服务器需要实现为多线程守护进程。

编译工具建议使用：

* `gcc`
* `make`
* `pthread`

编译时需要链接 pthread：

```bash
gcc ... -lpthread
```

或在 `Makefile` 中添加：

```makefile
LDFLAGS += -lpthread
```

## 4. 总体系统架构

系统由服务器、普通客户端和机器人管理器组成。

### 4.1 服务器 `chatserver`

服务器负责系统核心逻辑，主要功能包括：

1. 读取配置文件；
2. 根据配置文件创建公共 FIFO；
3. 创建日志目录和日志文件；
4. 创建固定大小线程池；
5. 将自身 daemonize，作为守护进程运行；
6. 主线程使用 `poll()` 监听 4 个公共 FIFO；
7. 收到请求后将请求分派给线程池中的空闲 worker；
8. 维护注册用户表；
9. 维护在线用户表；
10. 维护离线消息表；
11. 处理普通用户注册、登录、发消息、退出；
12. 处理机器人用户注册、登录、发消息、退出；
13. 记录服务器运行日志；
14. 记录线程池调度日志。

服务器主线程只负责监听请求和分派任务，不直接处理具体业务。具体业务逻辑应放在 `handlers.c` 中。

---

### 4.2 普通客户端 `client`

普通客户端用于模拟真实用户，主要功能包括：

1. 向服务器发送注册请求；
2. 向服务器发送登录请求；
3. 登录后创建用户专用 FIFO；
4. 从用户专用 FIFO 接收消息；
5. 向其他用户发送消息；
6. 向服务器发送退出请求。

普通客户端和服务器之间通过公共 FIFO 发送请求，通过用户专用 FIFO 接收服务器返回的信息和其他用户消息。

---

### 4.3 机器人管理器 `bot_manager`

机器人管理器负责动态管理机器人用户，主要功能包括：

1. 动态增加 `x` 个机器人；
2. 为每个机器人随机生成用户名和密码；
3. 每个机器人像普通客户端一样注册；
4. 每个机器人像普通客户端一样登录；
5. 每个机器人创建自己的用户专用 FIFO；
6. 每个机器人监听自己的专用 FIFO；
7. 收到普通用户消息后自动回复：

```text
幸会，用户名，很高兴认识您
```

8. 支持随机减少 `x` 个机器人；
9. 被减少的机器人向服务器发送退出请求；
10. 机器人相关行为也必须被服务器记录日志。

机器人管理器应作为特殊客户端实现，不应把机器人逻辑硬编码进服务器核心逻辑。

## 5. 配置文件要求

配置文件路径建议为：

```text
config/server.conf
```

配置文件至少包含以下内容：

```ini
SERVER_NAME=chatserver_ynp_1.0

FIFO_DIR=~/Server/fifo
REG_FIFO=ynp_reg_fifo
LOGIN_FIFO=ynp_login_fifo
MSG_FIFO=ynp_msg_fifo
LOGOUT_FIFO=ynp_logout_fifo

LOG_DIR=~/log/chat-logs/server
SERVER_LOG=server.log
THREAD_LOG=threads.log

POOLSIZE=100
```

其中：

* `ynp` 表示 `学生姓名拼音首字母缩写`；
* `your_name_pinyin` 表示 `学生姓名拼音`；
* 服务器命名格式为：

```text
chatserver_学生姓名拼音首字母缩写_x.y
```

其中：

* `x.y` 表示版本号；
* `x` 为偶数表示稳定版本；
* `x` 为奇数表示开发中版本；
* `y` 表示错误修补次数。

注意：C 程序中不能直接把 `~` 传给 `open()`、`mkfifo()`、`mkdir()` 等函数。程序需要通过 `getenv("HOME")` 展开用户主目录，或者在配置文件中使用绝对路径。

## 6. FIFO 设计要求

服务器需要创建 4 个公共 FIFO，统一存放在：

```text
~/Server/fifo/
```

4 个公共 FIFO 分别为：

```text
ynp_reg_fifo
ynp_login_fifo
ynp_msg_fifo
ynp_logout_fifo
```

含义如下：

| FIFO              | 用途        |
| ----------------- | --------- |
| `ynp_reg_fifo`    | 接收用户注册请求  |
| `ynp_login_fifo`  | 接收用户登录请求  |
| `ynp_msg_fifo`    | 接收用户发消息请求 |
| `ynp_logout_fifo` | 接收用户退出请求  |

每个客户端登录后需要创建自己的用户专用 FIFO，用于接收服务器返回的信息和其他用户发来的消息。建议命名为：

```text
~/Server/fifo/user_用户名_fifo
```

或：

```text
~/Server/fifo/用户名
```

客户端还可以创建临时回复 FIFO，用于接收注册、登录等一次性响应。建议命名为：

```text
~/Server/fifo/reply_进程号_fifo
```

## 7. 通信协议要求

通信协议优先使用固定长度 C 结构体，而不是复杂字符串解析。

建议在 `include/chat.h` 中定义统一请求结构体和响应结构体。

请求类型包括：

```text
REQ_REGISTER
REQ_LOGIN
REQ_MSG
REQ_LOGOUT
```

请求结构体应至少包含：

* 请求类型；
* 用户名；
* 密码；
* 发送者；
* 接收者；
* 消息内容；
* 客户端进程号；
* 客户端回复 FIFO；
* 用户专用 FIFO；
* 是否为机器人用户。

响应结构体应至少包含：

* 请求是否成功；
* 返回信息；
* 时间戳；
* 发送者；
* 接收者；
* 消息正文。

固定长度结构体便于 FIFO 读写，减少解析复杂度，适合作为本实验的通信协议。

## 8. 守护进程要求

服务器启动后需要转为守护进程。守护进程需要满足：

1. 通过 `fork()` 让父进程退出；
2. 子进程调用 `setsid()` 创建新会话；
3. 必要时再次 `fork()`，避免重新获得控制终端；
4. 修改工作目录；
5. 设置 `umask`；
6. 关闭标准输入、标准输出、标准错误；
7. 将标准描述符重定向到 `/dev/null`；
8. 后台常驻运行。

实验报告中需要通过 `ps` 命令查看服务器进程状态，说明服务器是守护进程。

建议使用命令：

```bash
ps -ef | grep chatserver
ps -o pid,ppid,stat,tty,cmd -C chatserver_ynp_1.0
```

说明要点：

* `TTY` 为 `?` 表示进程没有控制终端；
* `PPID` 通常为 1 或 systemd 相关进程，说明原父进程已退出；
* `STAT` 为 `S`、`Ss` 等状态，说明服务器作为后台进程等待请求。

## 9. 多路复用监听要求

服务器主线程需要通过 `select()` / `poll()` / `epoll()` 监听 4 个公共 FIFO，优先使用 `poll()`。

监听对象包括：

1. 注册 FIFO；
2. 登录 FIFO；
3. 发消息 FIFO；
4. 退出 FIFO。

建议打开 FIFO 时使用：

```text
O_RDWR | O_NONBLOCK
```

这样可以避免 FIFO 没有写端时读端反复遇到 EOF 的问题。

`poll()` 监听逻辑为：

1. 创建 `pollfd` 数组；
2. 将 4 个公共 FIFO 的 fd 放入数组；
3. 每个 fd 的 `events` 设置为 `POLLIN`；
4. 调用 `poll()` 阻塞等待；
5. 某个 FIFO 可读时，对应 `revents` 出现 `POLLIN`；
6. 读取请求结构体；
7. 将请求交给线程池处理；
8. 进入下一轮监听。

---

## 10. 线程池要求

服务器启动时需要创建固定大小线程池，线程数量由配置文件中的 `POOLSIZE` 决定，默认要求为：

```text
POOLSIZE=100
```

线程池需要满足：

1. 启动时预创建 100 个 worker；
2. 每个 worker 初始状态为 `IDLE`；
3. 请求到达时，从空闲 worker 中选择一个处理；
4. worker 处理请求时状态变为 `BUSY`；
5. 请求处理完成后 worker 回到 `IDLE`；
6. 刚回收的 worker 倾向于下一次优先被分派；
7. 线程创建、分派、处理、回收、状态变化都需要写入 `threads.log`。

建议设计：

* 使用固定大小 worker 数组保存所有线程；
* 使用 `idle_stack` 保存空闲 worker 编号；
* 分派时从栈顶弹出 worker；
* 回收时把 worker 压回栈顶；
* 这样刚回收的 worker 会优先被下一次分派；
* 线程池内部使用 `pthread_mutex_t` 和 `pthread_cond_t` 保证线程安全；
* worker 线程循环等待任务，不应每次请求都新建线程。

线程池日志路径：

```text
~/log/chat-logs/server/threads.log
```

## 11. 用户数据管理要求

用户数据、在线表、离线消息表由 `user_store` 模块统一维护。

建议在 `user_store.c` 中维护：

1. 注册用户表；
2. 在线用户表；
3. 离线消息表；
4. 用户登录状态；
5. 用户专用 FIFO 路径；
6. 是否机器人用户；
7. 离线消息 pending / sent 状态。

所有共享数据结构必须加 mutex，避免多个 worker 同时访问时出现竞态条件。

至少需要保护以下操作：

* 注册用户；
* 查找用户；
* 校验密码；
* 设置用户在线；
* 设置用户离线；
* 获取在线用户列表；
* 保存离线消息；
* 推送离线消息；
* 删除或标记离线消息状态。

## 12. 普通聊天功能要求

### 12.1 注册

注册功能需要满足：

1. 用户提交用户名和密码；
2. 用户名必须唯一；
3. 如果用户名不存在，则注册成功；
4. 如果用户名已存在，则注册失败；
5. 服务器将注册结果返回给客户端；
6. 服务器记录注册日志。

注册测试流程：

1. 启动 3 个终端；
2. 依次注册：

```text
学生姓名拼音_1
学生姓名拼音_2
学生姓名拼音_2
```

3. 第三个用户因为用户名重复，注册失败。

---

### 12.2 登录

登录功能需要满足：

1. 用户提交用户名和密码；
2. 服务器校验用户是否存在；
3. 服务器校验密码是否正确；
4. 登录成功后加入在线用户表；
5. 服务器返回当前在线总人数和在线用户名；
6. 登录失败时返回失败原因；
7. 登录成功后，服务器主动推送该用户的离线消息；
8. 服务器记录登录日志。

登录测试流程：

1. 启动 3 个终端；
2. 依次登录：

```text
学生姓名拼音_1
学生姓名拼音_2
学生姓名拼音_3
```

3. 前两个用户登录成功；
4. 第三个用户因为未注册，登录失败。

---

### 12.3 在线消息发送

发消息功能需要满足：

1. 用户 A 给用户 B 发送消息；
2. 如果用户 B 在线，服务器立即将消息转发到用户 B 的专用 FIFO；
3. 服务器记录消息发送日志；
4. 日志状态为 `sent`；
5. 如果连续成功发送信息超过 5 次，可在目标用户名后添加 `*` 表示重要朋友。

在线消息测试流程：

`学生姓名拼音_1` 给 `学生姓名拼音_2` 连续发送 5 次：

```text
How are you，学生姓名拼音_2
```

`学生姓名拼音_2` 的客户端应显示 5 次收到的消息。

---

### 12.4 用户退出

退出功能需要满足：

1. 用户向服务器发送退出请求；
2. 服务器将该用户从在线用户表移除；
3. 服务器向剩余在线用户广播该用户退出的信息；
4. 服务器向剩余在线用户广播新的在线用户列表；
5. 服务器记录退出日志。

退出测试流程：

`学生姓名拼音_2` 退出后，`学生姓名拼音_1` 能看到：

1. `学生姓名拼音_2` 已退出；
2. 当前剩余在线用户列表。

---

### 12.5 离线消息保存和推送

离线消息功能需要满足：

1. 如果接收者离线，服务器不丢弃消息；
2. 服务器将消息保存为离线消息；
3. 离线消息记录内容包括：

   * 发送者；
   * 接收者；
   * 消息内容；
   * 发送时间；
   * 状态 `pending`；
4. 接收者重新登录后，服务器主动推送离线消息；
5. 推送时需要展示原始发送时间；
6. 推送成功后记录日志，状态为 `sent`。

离线消息测试流程：

1. `学生姓名拼音_2` 退出；
2. `学生姓名拼音_1` 给离线的 `学生姓名拼音_2` 发送：

```text
Hi，let’s play badminton?
```

3. 服务器保存为离线消息；
4. `学生姓名拼音_2` 重新登录；
5. `学生姓名拼音_2` 收到离线消息，并看到发送时间。

## 13. 机器人管理器要求

机器人管理器需要支持两个核心操作：

```bash
./bot_manager add x
./bot_manager del x
```

其中 `x` 为机器人数量。

### 13.1 增加机器人

增加机器人时需要：

1. 随机生成机器人用户名；
2. 随机生成机器人密码；
3. 机器人像普通用户一样向服务器注册；
4. 机器人像普通用户一样向服务器登录；
5. 机器人登录后创建自己的专用 FIFO；
6. 机器人出现在在线用户列表中；
7. 普通用户可以看到在线机器人；
8. 机器人相关注册和登录操作写入服务器日志。

机器人用户名建议格式：

```text
bot_ynp_时间戳_随机数
```

---

### 13.2 机器人自动回复

机器人登录后应监听自己的用户专用 FIFO。

收到普通用户消息后，机器人自动向发送者回复：

```text
幸会，用户名，很高兴认识您
```

例如收到 `学生姓名拼音_1` 的消息后，回复：

```text
幸会，学生姓名拼音_1，很高兴认识您
```

机器人自动回复也应通过服务器的公共消息 FIFO 发送，不应绕过服务器直接写对方 FIFO。

---

### 13.3 减少机器人

减少机器人时需要：

1. 从当前在线机器人中随机选择 `x` 个；
2. 被选中的机器人向服务器发送退出请求；
3. 服务器将机器人从在线用户表中移除；
4. 服务器向在线普通用户广播机器人退出信息；
5. 普通用户可以看到机器人退出；
6. 服务器记录机器人退出日志。

注意：机器人用户名是随机生成的，下次增加机器人时不一定是同一个身份。因此离线消息管理需要特殊处理：

* 不应长期保存发给离线机器人的消息；
* 如果目标用户是离线机器人，建议直接丢弃消息并记录日志；
* 普通用户的离线消息仍然需要正常保存。

---

## 14. 日志要求

### 14.1 服务器日志

服务器日志路径：

```text
~/log/chat-logs/server/server.log
```

`server.log` 至少记录：

1. 服务器启动时间；
2. 配置文件读取结果；
3. 公共 FIFO 创建结果；
4. 日志文件创建结果；
5. 线程池创建结果；
6. 用户注册事件；
7. 用户注册失败事件；
8. 用户登录事件；
9. 用户登录失败事件；
10. 用户退出事件；
11. 在线消息发送事件；
12. 离线消息 pending 事件；
13. 离线消息 sent 事件；
14. 机器人注册事件；
15. 机器人登录事件；
16. 机器人发送消息事件；
17. 机器人退出事件；
18. 错误信息。

日志内容应包含：

* 时间；
* 事件类型；
* 用户名；
* 接收者；
* 消息状态；
* 错误原因。

---

### 14.2 线程池日志

线程池日志路径：

```text
~/log/chat-logs/server/threads.log
```

`threads.log` 至少记录：

1. worker 创建；
2. worker 初始状态；
3. worker 被分派；
4. worker 开始处理请求；
5. worker 处理完成；
6. worker 被回收；
7. worker 状态变为 `IDLE`；
8. worker 状态变为 `BUSY`；
9. 分派时间；
10. 回收时间。

---

### 14.3 日志线程安全

日志模块必须使用 mutex 保证线程安全。

原因是多个 worker 可能同时写入日志，如果不加锁，日志内容可能交错，影响实验展示和调试。

建议提供统一日志接口：

```text
log_server(...)
log_thread(...)
```

业务模块不应直接操作日志文件。

## 15. 文件权限要求

服务器日志文件需要限制权限，建议权限为：

```text
0600
```

即：

* 当前用户可读；
* 当前用户可写；
* 当前用户不可执行；
* 同组用户无权限；
* 其他用户无权限。

创建日志文件时可使用：

```text
open(path, O_CREAT | O_APPEND | O_WRONLY, 0600)
```

实验中需要使用以下命令查看日志文件权限：

```bash
ls -l ~/log/chat-logs/server/server.log
```

## 18. 设计约束和实现倾向

本项目实现时应遵循以下设计倾向：

1. 通信协议优先使用固定长度 C 结构体，而不是复杂字符串解析；
2. 主线程只负责监听公共 FIFO 和分派请求；
3. 业务处理放在 `handlers.c`；
4. 用户数据、在线表、离线消息由 `user_store.c` 统一维护；
5. 所有共享数据结构必须加 mutex；
6. 日志写入必须加 mutex；
7. 线程池使用 LIFO 空闲栈；
8. 刚回收的线程优先被下一次分派；
9. 机器人管理器作为特殊客户端实现；
10. 不要把机器人逻辑硬编码进服务器核心；
11. 离线消息不应长期保存给随机生成的机器人用户名；
12. 服务器日志和线程池日志分开记录；
13. worker 线程循环复用，不要每次请求都创建新线程；
14. 代码需要便于截图和报告说明。

---

## 19. 非目标

本实验不要求实现以下内容：

1. 图形界面；
2. 网络 socket 通信；
3. 数据库存储；
4. 加密认证；
5. 复杂好友系统；
6. 群聊系统；
7. 文件传输；
8. 长期持久化用户数据库。

用户数据和离线消息可以使用内存数组或简单文件存储。为了降低实验复杂度，建议优先使用内存结构维护状态，重点展示系统调用、线程池、FIFO、多路复用和日志功能。
